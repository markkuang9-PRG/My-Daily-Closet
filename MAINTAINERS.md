# Maintainers

This file explains how maintainers should keep the repository healthy as outside contributors start to participate.

## Core rule

Do not optimize for issue count. Optimize for contributor success rate.

## Maintainer expectations

- respond quickly to first-time contributors;
- keep issue scopes narrow;
- prefer small pull requests to broad rewrites;
- reject unclear architectural drift early;
- keep roadmap, contributor tasks, and repository policies aligned.

## Triage workflow

1. Confirm whether an issue fits the active roadmap.
2. Add clear labels.
3. If it is a first-contribution candidate, make sure the scope is small and testable.
4. If the issue touches licensing, trademarks, security, or business terms, route contributors to the relevant policy docs before implementation starts.

## Review guidelines

- ask for focused changes instead of piling on side quests;
- protect `main` from rushed merges;
- prefer preserving product momentum over theoretical refactors;
- keep contributor feedback factual and specific.

## First contributor experience

For first-time contributors, maintainers should try to provide:

- confirmation that the issue is still open;
- one sentence on expected scope;
- one sentence on what would count as “done”;
- a quick reply after the first PR is opened.
