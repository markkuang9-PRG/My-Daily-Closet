# Security Policy

## Supported Security Reporting Path

If you discover a security vulnerability, do not open a public GitHub issue with exploit details.

Use one of these private routes instead:

1. GitHub private vulnerability reporting, if enabled for this repository.
2. Direct private contact with the project owner through GitHub.

Include:

- a clear description of the vulnerability;
- affected files, endpoints, or workflows;
- reproduction steps or a proof of concept;
- impact assessment;
- any suggested mitigation.

## Coordinated Disclosure

Please give the maintainers a reasonable opportunity to investigate and remediate before public disclosure.

The project owner may decide severity, remediation timeline, disclosure timing, and whether a report qualifies for acknowledgement.

## Out of Scope

The following are generally not treated as security vulnerabilities unless they create a direct, concrete exploit path:

- typo fixes or documentation mistakes;
- theoretical issues without a reproducible path;
- issues that require already-compromised credentials without a product-side weakness;
- reports against third-party services that are outside this repository's control.
