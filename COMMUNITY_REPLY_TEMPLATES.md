# Community Reply Templates

Use these when people reply to the project publicly or privately.

Keep replies short, direct, and scoped.

## 1. Someone says they are interested

Thanks. The fastest entry point is [CONTRIBUTOR_START_HERE.md](CONTRIBUTOR_START_HERE.md).

If you want a narrow first task, start with one of the issue drafts in [community/issue-drafts](community/issue-drafts). I recommend picking one focused issue rather than browsing the whole backlog first.

## 2. Someone asks where to start

Start here:

1. [CONTRIBUTOR_START_HERE.md](CONTRIBUTOR_START_HERE.md)
2. [GOOD_FIRST_TASKS.md](GOOD_FIRST_TASKS.md)
3. one live issue from the current first-wave set

For a first PR, keep it small and easy to verify locally with `npm run lint`, `npm run build`, and `npm run test`.

## 3. Someone wants a first issue recommendation

A good first scoped task is the weather fallback issue.

Goal: improve weather mapping and fallback text without changing the provider or redesigning the flow.

If that is not your area, the upload toast or AI parsing test issues are also good starting points.

## 4. Someone proposes a broad rewrite

Thanks. I want to keep the first contributor wave tightly scoped.

If you want to help, it would be better to start with one contained issue or open a proposal that explains user impact, scope, and tradeoffs before changing architecture.

## 5. Someone asks whether the project is active

Yes. The repository is actively maintained and `main` is kept working.

The project is currently focused on turning the prototype into a cleaner MVP, especially around Firebase reliability, AI response handling, performance, and UX polish.

## 6. Someone asks whether commercial rights are preserved

The code in the repo is AGPL-3.0-only.

Contribution, trademark, and commercial boundary documents are in:

- [CLA.md](CLA.md)
- [TRADEMARKS.md](TRADEMARKS.md)
- [COMMERCIAL-LICENSING.md](COMMERCIAL-LICENSING.md)

If the question is business-sensitive, continue privately rather than in a public issue.

## 7. Someone opens a low-quality feature request

Thanks for writing this up.

Before this can move forward, please narrow it to:

- the user problem;
- the smallest useful scope;
- what success would look like.

Right now I am prioritizing issues that fit the current MVP roadmap.

## 8. Someone submits their first PR

Thanks for the PR. This is the right kind of scope for the current phase.

I will review it against current roadmap fit, user impact, and whether it keeps the app stable.

## 9. Someone goes quiet after showing interest

No pressure. If you want to jump back in later, start from [CONTRIBUTOR_START_HERE.md](CONTRIBUTOR_START_HERE.md) and pick one small issue.

The project is being kept current, so starting narrow is still the best path.
