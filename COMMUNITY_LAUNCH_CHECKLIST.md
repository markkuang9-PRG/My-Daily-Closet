# Community Launch Checklist

Use this checklist before and during the first external contributor push.

## Before posting publicly

- [ ] `main` is green and current
- [ ] README reflects the current product state
- [ ] [CONTRIBUTOR_START_HERE.md](CONTRIBUTOR_START_HERE.md) is easy to find
- [ ] [GOOD_FIRST_TASKS.md](GOOD_FIRST_TASKS.md) matches the current codebase
- [ ] [ISSUE_BACKLOG.md](ISSUE_BACKLOG.md) is current
- [ ] At least 3 live GitHub issues are published
- [ ] Labels exist: `good first issue`, `help wanted`, `frontend`, `firebase`, `ai`, `performance`, `ux`
- [ ] You can reply quickly if someone comments or opens a PR

## Recommended first live issues

Publish these first:

1. Improve weather code mapping and fallback states
2. Let users edit generated resale copy before copying
3. Reduce initial bundle size with targeted code splitting
4. Add upload success and failure toasts instead of `alert`
5. Add tests for outfit/result parsing edge cases

## After posting publicly

- [ ] Reply to interested contributors within 24 hours
- [ ] Point every new contributor to one narrow issue, not the whole backlog
- [ ] Keep at least 1 `good first issue` open
- [ ] Close or rewrite stale issues quickly
- [ ] Do not let unanswered PRs sit

## What not to do in the first wave

- [ ] Do not open 15 issues at once
- [ ] Do not invite broad rewrites
- [ ] Do not pitch acquisition or buyout outcomes in public recruitment posts
- [ ] Do not leave roadmap and issue scope mismatched
